# i18n.md — C-I18N (multi-language report — post-generation translation overlay)

Emits **`report.<lang>.html`** — a self-contained, per-language copy of the unified report
(`report.md` C-REPORT) in a chosen language. **English is the canonical analysis output and is
never touched** (`report.html`); a translated report is a **derived overlay**, so the grounded,
verified analysis can never be corrupted by translation. Read **`report.md` + `redaction.md` first**
(same artifact + redaction discipline). (S-8 / R9 untouched — a translated report is still a
deep-tier `.ai/` reader.)

## The 2 shipped languages (+ `other:<language>` for any other)
The single source of truth for the shipped set is `tools/build_i18n.py` `LANGS` (mirrored by the
config-schema `translate` enum + the template `STRINGS` table + the *Translate picker*). The set is
curated/chrome-baked/harness-tested down to Spanish + Hebrew (Hebrew exercises the RTL path):

| code | language | dir |
|------|----------|-----|
| `es` | Español | ltr |
| `he` | עברית | **rtl** |

`en` is the canonical base. **Any other language still translates on demand** via the
**`other:<language>`** escape hatch — content prose is translated, the page direction flips for a
known RTL script, and chrome falls back to English (**stated**, never silent). Trimming the *shipped*
set to two does NOT remove the capability (expand-only): the builder accepts any language code.

## Two string layers (the WHERE)
- **Chrome** (UI labels — Docs/Insights/nav/KPIs/card titles/state messages): a **static
  `STRINGS[lang]` table baked into `report-template.html`**, curated ONCE by humans and shipped for
  all languages (free at build time). The contract is `build_i18n.CHROME_KEYS` — every key present
  for every `LANGS` code (harness G0). Picked client-side by `<html lang>`.
- **Content** (per-repo prose — component roles/docs, decision rationale, issue claims, the
  overview/architecture, KL, open questions): translated by an **LLM pass** (the skill) into a
  committed **translation memory**; the deterministic builder consumes it.

## The deterministic / harness-testable boundary (WHEN = post-generation)
```
build_report.build_report_model(out_dir)         # canonical English (untouched)
  + translation_memory.json (LLM-produced)        # content-hash-keyed, committed
        └─ build_i18n.apply_tm(model, tm, lang)    # PURE: translate PROSE fields, honest-degrade
        └─ bdv.render(...) + _set_html_lang(...)   # same byte-stable render + <html lang dir>
        → report.<lang>.html
```
The **skill PRODUCES** the translation memory; **`tools/build_i18n.py` only CONSUMES it** and is a
PURE function of (model, TM, lang) — **no network, no LLM, no clock** (the harness pins it byte-for-
byte). The harness tests the BUILDER + the mechanism, never translation QUALITY (the same way the
English prose quality is never gated — chrome is human-reviewed, content honest-degrades).

## Grounded-token preservation (the smart value — `tools/i18n_tokens.py`)
Translate ONLY human prose; never the grounding. Before the LLM sees a string, `mask()` swaps every
protected span for an `⟦N⟧` sentinel; `restore()` puts the exact span back; `verify()` asserts every
token survives **verbatim** in the result. Protected: **code** (fenced + inline), **`file:line`
citations**, **record IDs** (`BR-/WF-/IP-/ADR-/KL-/ISS-/IF-`), and **product nouns**
(`DeepInit`, `DeepMap`, `Deep Fusion Labs`, `CLAUDE.md`, `AGENTS.md`, `.ai/`). A glossary keeps
domain terms consistent. **A miss or a corrupted token → the field HONEST-DEGRADES to English
(never blank, never fabricated — R1 / KL-learning:001).**

## Translation memory — `.ai/i18n/translation_memory.json` (skill writes, builder reads)
Content-hash-keyed so re-runs don't re-pay and the output is byte-stable:
```jsonc
{ "schema": "deepinit-i18n/tm/v1",
  "prompt_version": "i18n/2026-06",          // bump ⇒ all keys invalidate (re-translate)
  "glossary_hash": "sha256:…",               // part of every content key
  "model": "claude-opus-4-8",
  "entries": {
    "<content_key>": {                        // i18n_tokens.content_key(prompt_version|lang|glossary_hash|source)
      "lang": "he",
      "translated": "…(the final translated prose; every protected token verbatim)…",
      "tokens": ["src/auth/login.ts:42", "BR-auth:003", "DeepInit"] } },
  "glossary": { "issue": "…", "component": "…", "decision": "…" } }
```

## RTL (Hebrew `he`; future `fa`/`ur`)
`build_i18n` sets `<html lang="he" dir="rtl">`; the template uses **logical CSS** (`*-inline-start/end`)
+ `[dir="rtl"]` overrides (the left rail mirrors to the right; directional icons flip) and **isolates**
code / `file:line` / IDs as LTR runs (`direction:ltr; unicode-bidi:isolate`) so grounding never mirrors.
Metric **labels** are translated chrome; metric **values** are structural (never translated).

## Command / flag / config surface
- **`/deep-init:translate [<lang>]`** (`commands/translate.md`): with no language it opens the
  **Translate picker** (SKILL.md) — an AskUserQuestion with the top-4 as buttons + "Other", which
  expands to a numbered menu of the rest + "type any other language"; then builds/refreshes the
  English report, runs the content translation pass into the TM, and emits `report.<lang>.html`.
- **`--translate=<lang>`** flag + the `.ai/deepinit.config` `translate` key (enum of the shipped
  codes `es`/`he`, or `other:<language>` for any other; precedence defaults → config → flags).

## Verification
`tools/build_i18n.py --lang <code>` (or `all`) → opens each `report.<lang>.html` from `file://`:
chrome translated, `he` is `dir="rtl"` with the rail mirrored and code/`file:line`/IDs LTR-isolated,
protected tokens byte-identical to English, missing TM entries shown in English (never blank), build
twice → byte-identical. Pinned by harness **§71** (token round-trip, content-key, apply_tm
translate+honest-degrade+determinism, `<html lang dir>` incl. RTL, no-network boundary, chrome
completeness) + the jsdom smoke test, with a killing mutation per gate.
