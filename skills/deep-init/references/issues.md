# issues.md — C-ISSUE / the issue layer (detect → raise → verify → emit)

The issue layer turns DeepInit's already-extracted, grounded findings into a ranked, **report-only** list of the real problems in a codebase. It is an **extension of existing stages, not a new code-reading pass** (S-1): it operates over the BR/IP/WF/ADR sets and the horizontal coupling/drift outputs the pipeline already produced — never a fresh file read. Five families: **IF-1** business-rule violations, **IF-2** DB-vs-code drift, **IF-3a** silent cross-component coupling, **IF-4** intent/decision contradictions, **IF-5** risk-hotspot overlay + change-coupling.

**Read `global-rules.md` first.** This stage assumes R1 (never fabricate / cite + certainty), R7 (DB read-only gate), R9 (non-obviousness governs placement). New ID type: `ISS-{comp}:{nnn}` / `ISS-{nnn}` (R5).

## Where this sits in the pipeline
```
Detect → Plan → Extract → [Review*] → Filter ─┐
                                              ├─→ C-ISSUE (this file: detect candidates)
   (over already-extracted findings + horizontal outputs; never a fresh code read)
                                              └─→ C-RAISE (issue-filter.md: raise/suppress, the issue sibling of Filter)
                                                    → Verify (verification.md) → Redact (redaction.md) → Emit (generation.md: issues.md ledger + SARIF + dashboard)
```
*(The issue stream consumes already-extracted findings, so it is **order-independent of the doc-stream Filter** — the fork point above is illustrative; SKILL.md's run flow lists the issue pass alongside/just before Filter. Within the issue stream the canonical order is **Verify → Redact → Emit** — redaction is the final unconditional gate before anything touches disk, per the numbered flow below + `issue-filter.md`.)*
- **Scope:** IF-2/IF-5 are deterministic (consume already-computed signals — ~0 marginal tokens). The semantic families (IF-1/IF-3a/IF-4/IF-7(a)) reason over the extracted sets **scoped per-component + the existing horizontal pass — never all-pairs / file²**, so cost scales with component count. The issue-pass token estimate is added to the Detect cost preflight (FL-4); `--issues=off` skips the layer entirely and zeroes that estimate.

## Governing principles of the issue layer (non-negotiable)
- **Report-only.** The issue layer never modifies code (preserves the v2.0 read-only contract; AC-11). It writes only `.ai/` outputs + the lean pointer (owned-region).
- **Polarity flip — THE false-positive control (AF-1).** The issue layer biases toward **suppression**, the *opposite* of `redaction.md`'s "when in doubt, redact." For issues the asymmetry inverts: **a false-positive issue is the costly, trust-destroying outcome; a missed low-severity issue is tolerable.** Inherit `filter.md`'s "when in doubt keep in deep" bias **plus** an explicit FP-guard (the R1.5 back-to-code validate, see `issue-filter.md`) — never redaction's bias-toward-flagging.
- **Flag-don't-assert.** Wording is always tentative ("possible unenforced rule on path X"), never "this IS a bug." DeepInit **flags likely**, it does not prove correctness (verification checks citation-existence + plausibility, not correctness — scope-honesty, D2-019). Not a security product (IF-1(d) is framed as a business-rule violation, never "security"; DeepGuard owns security — S-11).
- **R9 — issues never enter the lean tier.** All issues live in the deep ledger (`.ai/docs/issues.md`), the dashboard, and SARIF. The lean `AGENTS.md` gets at most a one-line "Where to look → issues" pointer in its existing "Where to look" index. (S-3 / AC-10.)
- **Grounding is mandatory.** Every issue carries ≥1 `file:line` provenance + a certainty (R1) and is routed through `verification.md`; an issue whose citation does not resolve is dropped from any surfaced tier or flagged `[unverified]` in deep — **never surfaced as verified** (AC-6).
- **R7 for IF-2.** The live-DB half of IF-2 is behind the read-only DB gate. If the DB is skipped or the connection refused, IF-2 **suppresses** the live-drift checks (it never speculates a drift from the ORM side alone).

## Issue record — the ledger row (§4.1)
```
ISS-{comp}:{nnn}                # or ISS-{nnn} for cross-component (R5)
  family:        IF-1 | IF-2 | IF-3a | IF-4 | IF-5    # open enum; promoted roadmap families add IF-8 | IF-3b | IF-7 (c-slice + a-commission semantic) | IF-6 | IF-10 (see below; IF-9 deferred — semantic-only)
  claim:         one-line, flag-don't-assert description of the potential issue
  provenance:    [file:line, ...]            # ≥1; verified by verification.md
  severity:      Critical | High | Medium | Low      # ODC-style: behaviour impact, not linter noise
  criticality:   Core | Supporting | Peripheral      # lifted from extraction (do not re-derive)
  priority:      derived = f(severity, criticality, IF-5 signals)   # ranking, see IF-5
  certainty:     HIGH | MEDIUM | LOW          # R1
  non_obvious:   true                          # raise-vs-suppress gate; linter-territory candidates are suppressed, never stored
  verified:      true | false                  # verification.md
  verified_at:   {ISO}
  lifecycle:     open | resolved | regressed   # §"Lifecycle" below
  baseline:      new | persisting | accepted | resolved   # baseline diff (CD-E2)
  first_seen_run:{RUN_ID}
  sarif_rule_id: deepinit/IF-1 …              # SARIF mapping (CD-E1)
  cross_model_agreement: (unset)               # reserved for AgentAlliance — SAME field name + honesty framing as verification.md (D2-019); never invent a parallel field
```
**Severity** is ODC-style behaviour-impact (a Core-entity unenforced write is High/Critical; cosmetic ≈ Low), **never** a borrowed vendor severity. **Raise only Core/Supporting**; Peripheral findings are listed in the deep tier only, never ranked up. **priority** ranks both the top-risk zones and every IF-1…IF-4 issue by the priority of its location (IF-5); mirror `filter.md`'s lean ranking, not a churn-only sort.

## Baseline match key + normalization — the shared primitive (§4.2 / §4.4)
Both the **baseline diff** (T5.1) and the **SARIF `partialFingerprints`** (T4.2) key off ONE function — define it once here; never re-derive per consumer, and **never** derive it from the `content_hash`/`interface_hash` SHAs (those are unrelated; that confusion is an explicit lift-note).
```
match_key = ( family , normalized_location )

normalized_location:
  • resolve provenance file:line → its ENCLOSING SYMBOL (function/class/method) via
    structural-graph.json (Graphify) → ctags → grep fallback (R6/R8 ladder);
  • key on (relative_file_path, enclosing_symbol, intra-symbol ordinal),
    NOT the raw line number — so a line-shift-only refactor does NOT re-key the issue;
  • on SYMBOL_MOVED (reuse verification.md Pass-1 re-resolution), re-resolve to the
    moved symbol and keep the same key; scope re-keying to the DP-1 blast radius so an
    untouched component's issues never churn.
```
NASA-SCRUB-style match: *same family + same normalized location = the same finding* across runs. This is what lets the baseline suppress accepted FPs and surface only genuinely new issues, and what lets a downstream consumer (e.g. GitHub code scanning) baseline on the identical fingerprint.

## Lifecycle state machine (§4.2)
```
        (first detected, not in baseline)
                  │
                  ▼
   new ──accept (user / --issues-baseline=accept)──► accepted ──(reappears)──► persisting (suppressed from "new")
    │
    │ persists across runs
    ▼
  open ──(source changed; issue no longer detected AND re-verified gone)──► resolved
    ▲                                                                            │
    └────────────────(re-detected after being resolved)───────────────────── regressed
```
Driven by the **baseline diff** (CD-E2) on each `--update`/heal run, using the match key above. `resolved`/`regressed` transitions are appended to a dedicated **issue section** of `changelog.md` (FL-5) — not mixed into BREAKING/MODIFIED. Acceptance is user-driven and write-once-per-id: no auto-accept on first run; a reappearing resolved issue is `regressed` against the ORIGINAL accepted baseline, never silently re-accepted.

## Detect → raise → verify → emit (per-issue flow)
1. **C-ISSUE** (the per-family detectors below) emits candidate issue records over the extracted findings: provenance + provisional severity/certainty, `non_obvious` set, `flag-don't-assert` claim.
2. **C-RAISE** (`issue-filter.md`) annotates `raise|suppress`: Test-1′ linter-territory → suppress; Test-3 root-cause dedup (keep the most specific anchor); Test-2′ → severity; **and forces the R1.5 back-to-code validate for every raised semantic issue (IF-1/IF-3a/IF-4) regardless of review mode** (the FP-guard must not be disabled by `fast`). Suppressed candidates are dropped, never stored.
3. **`verification.md`** verifies provenance (Pass-1 existence + Pass-2 plausibility) + stamps certainty/`verified_at`; unresolved citation → drop / `[unverified]`. (IF-2 asymmetry: the ORM-side `file:line` is Pass-1 verifiable; the DB object name is R7-gated live-read provenance, not citation-verifiable.)
4. **`redaction.md`** runs unconditionally over all issue content before it touches disk (ledger, SARIF, and the dashboard's embedded JSON).
5. **`generation.md`** emits the deep ledger `.ai/docs/issues.md` + `deepinit.sarif` + the dashboard, and adds only the one-line lean pointer (R9).

## Detectors (per family)
*Per-family detection logic is specified below, each appended by its build task — IF-2 (T1.1), IF-5 (T1.2), IF-4 (T2.1), IF-3a (T2.2), IF-1 (T2.3). Each detector emits candidate records in the schema above and hands them to C-RAISE.*

### IF-2 — DB-vs-code drift defects (deterministic; consumes existing drift output)
**Inputs (CONSUME, never re-derive):** the field/type **ORM-drift table** from `database.md` ("ORM-drift diff" — which *already assigns a Severity column*) + the system-wide **Schema Drift Report** aggregated in the horizontal `data-layer.md`. IF-2 **promotes each drift row to an `ISS-` record**; it does not re-run schema reconciliation and **does not re-derive severity** (lift `database.md`'s Severity verbatim). The system-wide aggregation is the feed for cross-component ranking.

**Checks promoted (each already a drift row):**
- a column/table/field the code references but **absent in the live schema** (model-only field) — certainty HIGH (genuinely missing); severity per `database.md` (model-only field = High);
- a DB **NOT-NULL / UNIQUE / FK** constraint not honored by writing code;
- an **enum / CHECK-constraint** mismatch code↔DB;
- a query on an **unindexed column on an implied hot path** (the one perf-ish check kept).
- Orphan table (table with no model) → informational, Low.

**Type-equivalence normalization (NEW — the prime FP source; BASE-TYPE-ONLY):**
- Normalize dialect **synonyms of the base type only**: Postgres `decimal`≡`numeric`, `int4`≡`integer`, `bool`≡`boolean`; MySQL `tinyint(1)`≡`boolean`; `varchar`≡`character varying`.
- **PRESERVE precision / scale / length / nullability comparison.** `decimal(10,2)` vs `numeric(12,4)` **STILL raises** (params differ → precision mismatch); only a bare-synonym pair with *identical* params is suppressed. *(Critic fix: a naive "decimal==numeric" rule would false-negative the already-validated `mini-orm-drift` precision drift on `total`/`discount_rate` — those MUST still fire; assert it in the regression gate.)*

**R7 gate:** the live-drift checks require an **actual R7-approved live read**. If the DB is `--skip-db`, undetected, or the connect is refused, IF-2 **suppresses** the live checks entirely — no speculation from the ORM side alone. (Asymmetry: the ORM-side `file:line` is Pass-1 verifiable; the DB object name is R7-gated provenance, not citation-verifiable.)

**Framing:** "observed/likely divergence between code and the live schema," never "this will fail at runtime." Certainty HIGH where a referenced column genuinely does not exist; FP LOW (checkable).

**Test (AC-2):** `mini-if2-drift` fixture — ORM field absent in live DB → IF-2 High at the ORM `file:line` with severity lifted from `database.md`; a dialect-equivalent pair with identical params does NOT raise; the existing `mini-orm-drift` precision/type drifts STILL fire (regression guard); the DB-skipped path emits no IF-2.

### IF-5 — Risk-hotspot prioritization overlay + change-coupling (deterministic; ranking, not a bug family)
**Inputs (all reused — ~0 marginal tokens):** git-intel from `detection.md` (churn 6mo, bus-factor, tech-debt commit signal) + the **change-coupling** signal (new, computed in `detection.md`) + **Legacy Health Flags** (`discovery.md` §7: no-tests / no-docs / high-churn / single-author) + **criticality** (Core/Supporting/Peripheral, from `extraction.md`) + missing-test + the horizontal `technical-dependencies.md` cascade (`imported_by` closure, for impact breadth).

**Two jobs:**
1. **Rank every IF-1…IF-4 issue** by the priority of its location:
   `priority = f(severity, criticality[Core↑], churn↑, bus_factor[low↑], coverage[low↑])` — mirror `filter.md`'s lean ranking (behaviour-change first), **NOT a churn-only sort** (else git noise dominates). IF-5 sets each issue's `priority`; it does not raise standalone bugs from ranking alone.
   **Concrete weighting (the single source of truth — the harness asserts against THIS line, never an inline constant; change here ⇒ change the harness in lockstep):**
   `priority = 1000*CRIT + churn_6mo + (100 - coverage_pct) + (50 if bus_factor==1 else 0)`, where `CRIT = {Core:3, Supporting:2, Peripheral:1}` and only Core/Supporting issues are ranked. The dominant `1000*CRIT` term guarantees criticality outranks raw churn (a high-churn **Peripheral** file can never bury a **Core** one — the not-churn-only property); the remaining terms break ties by churn, then low coverage, then single-author bus-factor. Weights are illustrative-but-fixed.
2. **Surface hidden coupling** from change-coupling: a file pair co-changing ≥ threshold (default 3 shared commits / 6mo) with **no structural edge** → an IF-5 hidden-coupling issue (the co-change relationship is the "resource"; blast radius = the two components).

**Distinct from IF-3a — non-double-emit guard.** IF-5 change-coupling is the **temporal** signal (co-changed, no structural edge); IF-3a is the **static** signal (a NAMED shared resource read/written by ≥2 components with no interface). A **co-change-only** pair (no shared resource) surfaces as **IF-5 hidden-coupling ONLY**; a **shared-resource** pair surfaces as **IF-3a**; a pair that is **both** is emitted **once with a cross-reference**, never twice.

**Certainty / coverage:** deterministic, FP LOW. On thin git history (shallow clone / young repo — flagged by `detection.md`'s history-depth preflight), IF-5 outputs carry **`[LOW]`** and the run STATES the reduced coverage — never a fabricated hotspot or a silent zero.

**Test:** ranks issues by criticality×churn×bus-factor×coverage (not churn alone); change-coupling flagged on `mini-if5-cochange` at support ≥3; a co-change-only pair does NOT also emit IF-3a; thin-history → `[LOW]` + stated coverage.

### Class-conformance census — evidence/ranking OVERLAY on IF-4/IF-6/IF-7(a) (deterministic; NOT a detector)
*The one honest, structurally-clean artifact that fell out of the IF-9 convention-drift analysis (design-workshop + an independent adversarial-refutation panel, test-plan Run 13/14 — `tests-fixtures-v1/_external/_if9_convention_workshop_log.json`). IF-9 **as a detector** deferred (its outlier→defect step is intent-laden), but the **census** it would have computed clears the structural test as a **non-detector**: a pure deterministic structural count that asserts **no new defect predicate**, so it is subject to no intent-guess. It RIDES on an already-raised, already-validated host fire — it **never fires on its own, never changes whether a host fire happens, and never suppresses one** (provably zero recall/FP impact; additive-only).*
- **Trigger.** A raised **IF-4(a) / IF-6 / IF-7(a)** issue whose cited rule is **CLASS-RANGING** — its text quantifies over a class ("**every/all/each** `<class>` MUST `<property>`") where (i) the class maps to a **structurally enumerable** set of sibling sites (a glob / decorator / base-construct / directory role) and (ii) the required property is a **decidable structural check** (an AST `extends`/`implements` clause, a decorator presence, a method-exists). **Degrade-don't-guess (the membership guard).** The class enumeration **excludes generated / test / vendored paths** (`*.gen.*` · `*.test.*`/`*.spec.*` · `**/generated|vendored|__tests__/**` · `_pb2.py` · a `@generated` header) so codegen/test-doubles can't inflate or deflate the count; and a cardinality-1 decision (no class quantifier), a non-structural property ("cohesive", "idiomatic"), a non-enumerable class, **or a conformance property that cannot be a single unambiguous structural fact** (an **aliased / re-exported / namespaced base** like `extends repo.BaseRepository`, a multi-line `extends`, an interface-vs-class ambiguity) → **DEGRADE** (the overlay emits nothing rather than show a possibly-miscounted census; the same discipline as IF-6's non-enumerable guard).
- **Census + signal (deterministic).** Enumerate the class (N members), apply the structural conformance check → `conformers k`, `deviants = N-k` (each a real `file:line`, comment/string-blanked like §25/§29). Then `signal = f(N,k)`: **DEGRADE** if `N<3` (no majority can exist); **CORROBORATE** if `k ≥ ceil(2N/3)` and `k > N-k` (a strong majority conforms → the fired site is a clear minority outlier → annotate "lone deviant among N — k siblings conform" + raise the host fire's **IF-5 priority/rank** — *rank only, never certainty*, since the host family already set certainty); **STALE** if `N ≥ 4` and `k ≤ floor(N/3)` (the documented rule is contradicted by the **majority of its own class**). **STALE is a NEUTRAL rule-health FLAG, never an auto-judgment** — majority-deviation is genuinely ambiguous (the rule may be **de-facto stale** OR **systemically violated**; *5 of 6 services missing a mandated auth/audit base is a crisis, not a dead rule*), so the overlay **flags-don't-asserts**: it attaches a hedged caveat *"rule-health: k of N of the class also deviate — confirm de-facto-superseded vs systemically-violated"* and **changes NOTHING** (no priority, no certainty, no raise/suppress — the stale-vs-systemic call is the human's). The `N ≥ 4` floor keeps a thin 2-of-3 split out. **NEUTRAL** otherwise (attach the census, no change). **Pre-arithmetic qualitative override (gated by §32, finding A1 — so the signal is NOT (N,k)-deterministic):** force **DEGRADE** (never CORROBORATE/STALE), *regardless of the arithmetic*, when the host rule is a **superseded-contract** (the class-deontic was superseded by a broader runtime-asserted / enforced contract — even absent a `Status: Superseded` record; e.g. pyccel's literal one-base CHANGELOG rule superseded by a 3-base runtime `assert`, where (N,k)=(114,103) computes CORROBORATE yet DEGRADE is correct) or a **soft-rule** (advisory `should`-wording, not a hard MUST; e.g. eabe ADR-0008, (N,k)=(95,33)). The (N,k) arithmetic applies **only** to a live, hard, structural class-deontic; the controlled DEGRADE-guard vocab §32 gates is `{superseded-contract · soft-rule · non-structural · below-N3 · generated-or-ambiguous-membership}`.
- **Additive-only invariant (the honesty bar).** The overlay's only effects on a fire that already passed its host family's gate (incl. the forced R1.5 for the semantic hosts) are: **annotate evidence**, and — for **CORROBORATE only** — raise **IF-5 `priority`/rank** (never certainty). **STALE and NEUTRAL change nothing**; **DEGRADE** emits nothing. It NEVER raises a standalone issue, NEVER suppresses a fire, and NEVER changes whether a fire happens or down-ranks a Core/security fire (**provably zero recall/FP impact**, and no path to bury a systemically-violated rule — the adversarial-review fix, Run 14). It is **deterministic given the parsed (class, conformance, contract-status) inputs** (structural enumeration + structural conformance + threshold math + the §32 pre-arithmetic guard) and so **skips the forced R1.5 validate** like IF-5/§24/§25/§29 — there is nothing semantic left to confirm. The NL `rule → (class selector, conformance check)` parse is the **semantic** step (live-run, like IF-4's rule parse); the census **computation** is the deterministic core.
- **Distinct from IF-9 / non-double-emit.** This is **not** IF-9: IF-9-as-a-detector would fire on the outlier as a *new defect* (intent-laden, deferred); the census only **enriches/ranks** a fire another family already raised, and is silent unless such a host fire exists. It emits no `ISS-` of its own (no double-emit by construction). The STALE arm deliberately stops at *observation* — auto-asserting "the rule is de-facto superseded" would smuggle back the exact rule-level intent inference IF-9 deferred.
- **Test (deterministic, §31):** `mini-conformance-census` — ADR-100 repos (4 of 5 extend `BaseRepository`) → **CORROBORATE** the IF-4 fire on the lone `AuditRepository`; ADR-101 services (only 1 of 6 still extend the mandated `LegacyService`) → **STALE = neutral rule-health flag** (no down-rank); ADR-104 gateways (N=2) → **DEGRADE**; a `@generated` repo matching the glob → **excluded** (N stays 5); ADR-102 (cardinality-1) + ADR-103 (non-structural "cohesive") carry **no census_spec** (degrade-don't-guess). Gated by `_chat_validation.py` **§31** (10 checks; the threshold math, the inverted-conformance RED mutation, and the disabled-exclusion RED mutation are all load-bearing). Hardened by a 3-lens adversarial review (Run 14): the STALE arm was de-fanged from an auto-down-rank to a neutral flag, the membership guard + N≥4 floor added, and a conformer-grounding gate (G10) closes the "a counted conformer is never verified" gap.
- **Real-repo precision validation (Run 16, harness §32).** The guarded census + IF-4 were run against 4 pinned multi-stack OSS repos carrying a documented class-ranging rule (visdom CORROBORATE 18/13, kemal STALE 12/0, pyccel DEGRADE 114/103, eabe DEGRADE 95/33): the guarded path avoids the naive `mismatch=violation` detector's `N−k` false positives — **Σ 90 naive FPs avoided, 0 false defects**, every signal correct. Two **guard-divergent DEGRADEs** (pyccel superseded-contract; eabe soft-rule) prove the signal is *guarded, not purely arithmetic*. Records: `validation/results/*.json` (schema `deepinit-validation/precision-record/v1`), gated by `_chat_validation.py` **§32** (8 gates, RED-confirmed).

### IF-4 — Intent/decision contradictions (semantic; lowest-FP, most unique)
**Inputs:** ADR/KL + Design Rationale + Legacy Warnings + code. Reuses `review.md` R1 Consistency/Conflict contradiction types.
- **(a) ADR DRIFTED** — reuse `--update-adr`'s CONFIRMED / DRIFTED recheck (`adr.md` + `update.md`): code contradicts a recorded decision. When EXTENDED to BRs, use a **tighter, BR-specific drift definition** — a BR is a rule statement, not a decision-with-rationale, so "code moved away" is fuzzier and over-flags; require the code to *contradict* the rule, not merely diverge in style.
- **(b) Stale workaround** — a `WA-` whose original triggering condition no longer holds.
- **(c) Contradicts-naive-reading** — reuse `filter.md`'s flag: a name that lies, a "temporary" that is load-bearing, a cache that is the source of truth.
- **(d) Load-bearing TODO/FIXME** on a Core path (from the detection.md tech-debt grep).
- **Dual-citation (required for (a)/(b)):** the issue record's `provenance[]` MUST carry BOTH sides — the **decision reference** (the `ADR-`/`BR-`/`WA-` id **and** its `file:line` in `decisions.md`/the rule source) AND the **contradicting code** `file:line`. An IF-4(a)/(b) record citing only one side is incomplete — drop it rather than emit a half-grounded contradiction (R1 / AF-1).
- **Certainty** often HIGH (a concrete documented-vs-actual diff); FP LOW–MED.
- **Test (AC-2):** `mini-if4-adr-drift` — an ADR contradicted by current code → IF-4 citing BOTH the ADR and the contradicting `file:line`; a BR whose code legitimately *evolved* (no contradiction) does NOT over-flag DRIFTED. (Plus a stale `WA-`, a name-lie, and a load-bearing TODO on a Core path.)

### IF-3a — Silent cross-component coupling (semantic judgement over an existing graph)
**Inputs (PROMOTE existing horizontal output):** the `technical-dependencies.md` coupling (afferent/efferent, hubs, cascade = `imported_by` closure), `cross-references`/`domain-model` concept-ownership shadow copies, and the IP set's shared-DB type. IF-3a does not recompute the graph — it **judges** it.
- **Net-new judgement:** a DB table (or global / singleton / file / queue) **read/written by ≥2 components with no explicit interface between them** → silent coupling ("change one, break the other"). "**No explicit interface**" is checked against **all v2 edge classes** — `imports_from`/`imported_by`, the runtime `calls_into`/`called_by`, AND `inherits_from`/`inherited_by`: a pair sharing a resource with **no edge of any class** between them is the strongest silent-coupling signal (invisible at compile- AND run-time — the Rails/Zeitwerk case where there is no import edge at all yet the components are coupled through a shared table). Output **names the resource + the coupled components + the blast radius** (the `imported_by` closure, widened by `called_by`). Severity **High if the resource is Core**; FP LOW–MED (grounded in the shared-resource graph).
- **Static vs temporal (non-double-emit, see IF-5):** IF-3a = a NAMED shared resource (static). A co-change-only pair with no shared resource is **IF-5**, not IF-3a. A pair that is both → emitted once with a cross-reference.
- **Test (AC-3):** `mini-if3a-coupling` — a shared table written by two components with no interface → IF-3a names both + the table + blast radius; a documented-exception shared table (FP-trap) does NOT fire.

### IF-1 — Business-rule violations & inconsistencies (semantic; PRECISION-FIRST — the make-or-break family)
**Stance (user decision): Careful / avoid false alarms.** Raise only on a high-confidence, exact contrast; a missed subtle case is acceptable, a false alarm is not (the polarity flip above). **Inputs:** BR set (`rule | criticality | file:line`), cross-component BRs, IP/WF (to find sibling paths), code.
- **(a) Inconsistent enforcement** *(genuinely net-new inference — highest FP risk)*: a rule enforced on one path but absent on another path reaching the **same entity + operation**. **AF-1 FP-guard — raise ONLY when** (i) both paths write the **same entity+operation** (keyed on entity+operation, **not** fuzzy rule-text), (ii) the rule is **Core/Supporting**, (iii) **no alternative guard exists upstream** on the unguarded path. **The concrete contrast IS the evidence; with no guarded sibling to contrast against, do NOT raise.** Certainty starts **LOW → MEDIUM only when the contrast is exact**.
- **(b) Unenforced rule** — **PROMOTES** the existing `cross-references` "BR enforced by no workflow" coverage gap + `review.md` Missing-connections (half-lifted; not from-scratch).
- **(c) Contradictory rules** — two BRs on the same entity/field with conflicting conditions.
- **(d) Business-logic-security sub-class (un-branded, S-11)** — a write/operation path to a Core entity missing an authorization/ownership check that sibling paths have. Framed as an access-control **business-rule** violation, **never "security"** (DeepGuard owns security).
