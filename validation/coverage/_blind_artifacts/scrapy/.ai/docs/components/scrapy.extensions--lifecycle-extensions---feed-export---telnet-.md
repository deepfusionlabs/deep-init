# Component: scrapy.extensions (lifecycle extensions / feed export / telnet)

Path: `scrapy/extensions/`

Provenance: DeepInit EXTRACT stage, BLIND re-derivation from source only (no prose docs used). Every claim cites a file:line opened under `C:/tmp/p5_scrapy_blind/scrapy/extensions/` (or the manifest/registration files explicitly named).

## Role

- A collection of independent, optional, signal-driven add-on classes — each instantiated via a `from_crawler(cls, crawler)` factory that wires the object's handlers onto crawler signals (`crawler.signals.connect(...)`) and aborts construction with `NotConfigured` when its enabling setting is absent — that observe and react to the crawl lifecycle (close conditions, stats, throttling, memory, telnet, feed export). Pattern witnessed in `corestats.py:27-36`, `closespider.py:83-85`, `throttle.py:41-43`, `telnet.py:69-71`.
- The directory also hosts the feed-export subsystem (`FeedExporter` + pluggable storage backends + post-processing) `feedexport.py:444-451` and the HTTP cache policy/storage classes `httpcache.py:35,247,310`.

## Boundary rules

- Self-gating contract: an extension that should not run raises `scrapy.exceptions.NotConfigured` in `__init__`/`from_crawler` rather than loading inert — `closespider.py:58-59`, `throttle.py:24-25`, `memusage.py:37-38`, `telnet.py:44-45`, `feedexport.py:461-462`, `spiderstate.py:27-28`, `statsmailer.py:42-43`, `memdebug.py:30-31`.
- Construction contract: extensions are instantiated by the framework through `from_crawler(cls, crawler) -> Self` (the registration entrypoint) — uniform across `corestats.py:27`, `closespider.py:83`, `throttle.py:41`, `memusage.py:68`, `logstats.py:35`, `spiderstate.py:24`, `statsmailer.py:39`, `telnet.py:69`, `feedexport.py:445`, `logcount.py:32`, `memdebug.py:28`, `periodic_log.py:60`.
- Registration / layering: this component does NOT load itself — extensions are listed in `EXTENSIONS_BASE` and loaded by the framework's `ExtensionManager` which lives OUTSIDE this directory at `scrapy/extension.py:18` (`class ExtensionManager(MiddlewareManager)`); the base registry is `scrapy/settings/default_settings.py:344-355`. Feed storages/exporters are registered by scheme/format in `FEED_STORAGES_BASE` (`default_settings.py:376-382`) and `FEED_EXPORTERS_BASE` (`default_settings.py:363-372`); HTTP cache policy/storage in `default_settings.py:417-418`.
- Reactor-thread discipline: blocking storage I/O must not run on the Twisted reactor thread — `BlockingFeedStorage.store` hands work to `run_in_thread` and returns a `Deferred` — `feedexport.py:132-133`; FTP/S3/GCS implement `_store_in_thread` `feedexport.py:251,307,353`.
- Feed storage interface boundary: any custom storage MUST implement the `IFeedStorage` zope interface (`__init__(uri, *, feed_options)`, `open(spider)`, `store(file)`) — `feedexport.py:91-106`; a typing-only mirror `FeedStorageProtocol` exists for type hints `feedexport.py:108-121`.
- Input boundary on telnet: the telnet console requires a Twisted reactor and refuses to start without one — `telnet.py:47-51`.

## Key facts

- `EXTENSIONS_BASE` priorities are all `0` in the manifest (`default_settings.py:345-354`) — load order is not differentiated by priority here; the order is effectively registry insertion order, NOT a meaningful priority gradient (non-obvious: the value column is uniformly 0).
- The component is a flat package of unrelated classes: `scrapy/extensions/__init__.py` is EMPTY (0 lines, verified `wc -l`) — there is no package-level aggregation/re-export; each module is loaded by dotted path from the settings registry.
- AutoThrottle's core algorithm: target_delay = latency / TARGET_CONCURRENCY, new_delay = mean of old+target then clamped to [mindelay, maxdelay]; and it deliberately will NOT *decrease* delay on non-200 responses (error/redirect pages are small and would create a positive-feedback delay collapse) — `throttle.py:104-129` (esp. the guard `throttle.py:126-127`).
- Telnet console security: when `TELNETCONSOLE_PASSWORD` is unset, a random 8-byte hex password is generated per run and logged at INFO — `telnet.py:62-64`; auth is checked against username+password in a per-connection `Portal.login` `telnet.py:89-94`; the console exposes a live Python `manhole` over the engine/crawler/stats `telnet.py:96-122`.
- Spider shutdown is driven by two distinct extensions firing `engine.close_spider_async(reason=...)`: CloseSpider on count/timeout thresholds (`closespider.py:148-150`) and MemoryUsage when `MEMUSAGE_LIMIT_MB` is exceeded (`memusage.py:124-129`); reasons are stable strings (`closespider_itemcount`, `memusage_exceeded`, etc.).
- HTTP cache RFC2616 freshness logic ports Mozilla/Firefox heuristics (it cites nsHttpResponseHead source URLs) — `httpcache.py:195-196,228-229`; it caches 300/301/308 indefinitely (`httpcache.py:109-110,219-220`) and serves a stale cached response on a 5xx unless `must-revalidate` is set (`httpcache.py:165-173`).
- Persistence uses Python `pickle` (protocol 4) in three places — spider state (`spiderstate.py:39,44`), DBM cache (`httpcache.py:293,307`), filesystem cache meta (`httpcache.py:368,391`); the loads are explicitly marked unsafe (`# noqa: S301`), a security-relevant trust boundary on cache/jobdir contents.
- FeedExporter supports multi-file batched output: when `FEED_EXPORT_BATCH_ITEM_COUNT`/`FEEDS.batch_item_count` is set, it rolls to a new `FeedSlot` per batch and REQUIRES the feed URI to contain `%(batch_time)s` or `%(batch_id)d` (validated, else NotConfigured) — `feedexport.py:582-648,666-682`.
- Feed post-processing pipeline: `PostProcessingManager` subclasses `io.IOBase` so it can stand in as a writable file wrapping a reverse-ordered plugin chain (gzip/bz2/lzma compressors) between the exporter and the storage file — `postprocessing.py:115-166`, hooked in `feedexport.py:406-414`.
- `S3FeedStorage`/`GCSFeedStorage`/`boto3`/`google.cloud.storage` are LAZILY imported inside the storage classes and raise `NotConfigured("missing boto3 library")` if absent — these are optional, not hard dependencies — `feedexport.py:200-203,309-315`.
- `StatsMailer` and `MEMUSAGE_NOTIFY_MAIL` are deprecated — `statsmailer.py` emits a `ScrapyDeprecationWarning` at module import time (`statsmailer.py:25-30`) and memusage warns when the mail setting is used (`memusage.py:50-57`).
- `debug.py` (StackTraceDump / Debugger) and `memdebug.py` rely on POSIX signals (SIGUSR2/SIGQUIT) and gracefully no-op on win32 where those signals are absent — `debug.py:36-41,74-75`.
- The feed subsystem is signal-coupled both ways: it CONSUMES `spider_opened`/`spider_closed`/`item_scraped` and PRODUCES `feed_slot_closed`/`feed_exporter_closed` (custom signals it sends) — `feedexport.py:448-450,538,578-580` (signal objects defined in `scrapy/signals.py:27-28`).
