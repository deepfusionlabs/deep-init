## Boundary rules

- Zero-copy / borrow-only contract: parsers never allocate from or own the input — success returns a sub-slice of the original input and the remaining suffix, and failure returns the original input unchanged (helix-parsec/src/lib.rs:13-16; helix-parsec/src/lib.rs:78-82 `&input[self.len..]` / `&input[0..self.len]`).
- Pure library boundary: it is purely additive to the rest of the workspace (no outbound edges, no I/O, no globals); it can only be consumed, never reach back into another component (helix-parsec/Cargo.toml:14; absence of any `use` in helix-parsec/src/lib.rs).

## Key facts

- Result convention is the invariant the whole API is built on: success is `Ok((next_input, value))` and failure is `Err(input)` (the *unconsumed* input, not an error type), so combinators can backtrack by re-using the returned `Err` slice (helix-parsec/src/lib.rs:13-16; e.g. `or` retries `parser2` on the original `input` at helix-parsec/src/lib.rs:369-372).
- The central abstraction is the `Parser<'a>` trait with an associated `Output` type and a single `parse(&self, input) -> ParseResult` method (helix-parsec/src/lib.rs:33-37); a blanket impl makes any `Fn(&'a str) -> ParseResult<'a, T>` a `Parser`, so most combinators are written as closures returned from higher-order functions (helix-parsec/src/lib.rs:43-53).
- String literals are parsers: `impl Parser for &'static str` matches a literal prefix, but it deliberately conflicts with `str::parse`, so the `token` helper exists purely to call `.parse` on a bare literal (helix-parsec/src/lib.rs:74-83 impl; helix-parsec/src/lib.rs:60-63 + 101-103 the `token` work-around).
- Heterogeneous sequencing is implemented by `impl Parser for` tuples of sizes 2..=10, generated by the `seq_impl!` / `seq_body_impl!` macros, so a `seq!(...)` returns a tuple whose elements may each have a different `Output` type (helix-parsec/src/lib.rs:183-225; the `seq!` macro itself just builds a tuple at helix-parsec/src/lib.rs:176-181).
- `choice!` is variadic alternation built by recursively nesting binary `or` (all branches must share one `Output` type), distinct from `seq!`'s heterogeneous tuple (helix-parsec/src/lib.rs:241-249).
- Infinite-loop guard: the repetition combinators `zero_or_more` / `one_or_more` wrap their inner parser in `non_empty`, which fails if a parse consumed no input (`new_input.len == input.len`), preventing an endless loop on a zero-width match (helix-parsec/src/lib.rs:462, 495; helix-parsec/src/lib.rs:565-574 `non_empty`).
- Combinator inventory (the public surface): primitives `token`, `take_until`, `take_while`; combinators `map`, `filter_map`, `reparse_as`, `filter`, `or`, `optional`, `left`, `right`, `zero_or_more`, `one_or_more`, `sep`, `non_empty`; macros `seq!`, `choice!` (helix-parsec/src/lib.rs:101, 122, 144, 267, 290, 314, 335, 364, 390, 413, 434, 458, 491, 530, 565; macros at:176-181 and:241-249).
- Provenance / technology choice noted in source: the design follows the "bodil.lol parser-combinators" post for `sym`/`map`/`pred`/`one_or_more`/`zero_or_more`/`Parser`, with remaining combinators adapted from snippets.nvim's snippet parser or novel (helix-parsec/src/lib.rs:6-11) — i.e. a hand-rolled combinator library, not a third-party parsing crate (consistent with the empty `[dependencies]`, helix-parsec/Cargo.toml:14).
